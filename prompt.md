# 分析框架
请按以下结构逐步完成分析，使用链式思考（Chain-of-Thought）方法：

## 第一步：项目现状盘点
**输出格式：Markdown 表格**

| 功能模块 | 需求状态 | 实现进度 | 完成度 | 关键待办 |

| 例：字段类型定义 | 已定义 | 部分实现 | 60% | 缺少可变长度类型 |

**要求**：

## 第二步：需求合理性分析
**评估维度**：

### 2.1 业务价值评估
对每个需求进行四象限分类：
高价值 + 高频使用 → 核心需求（优先级：P0） 高价值 + 低频使用 → 重要需求（优先级：P1） 低价值 + 高频使用 → 便利需求（优先级：P2） 低价值 + 低频使用 → 可选需求（优先级：P3，建议移除）

### 2.2 需求冲突检查
- 是否存在互相矛盾的需求？
- 是否有过度设计（over-engineering）的功能？
- 哪些需求可以通过已有功能组合实现？

**输出格式**：
```markdown
### 需求 X：[需求名称]
- **价值评分**：[1-10分]
- **使用频率预估**：[高/中/低]
- **优先级建议**：[P0/P1/P2/P3]
- **合理性结论**：[保留/简化/移除/合并到其他需求]
- **理由**：[具体说明]

## 第三步：语法设计合理性分析
关键评估点：

3.1 表达能力检查
✅ 能否清晰描述所有目标二进制格式？
✅ 是否支持常见模式：定长/变长/条件字段/数组/嵌套结构？
❌ 是否存在无法表达的合理场景？
3.2 语法一致性
命名规范是否统一（如关键字大小写、标识符风格）？
类似功能的语法是否对称（如定义和引用的对应关系）？
是否有语法糖导致"多种方式实现同一目的"的混乱？
3.3 可读性评估
非专业用户能否在10分钟内理解基本语法？
示例代码是否自解释（self-documenting）？
是否需要大量注释才能理解代码意图？
输出格式：

Copy### 问题：[具体语法问题]
- **严重程度**：[高/中/低]
- **影响范围**：[解析器/用户体验/可维护性]
- **示例**：
  // 不良设计
  bad_example_code_here
  
  // 建议改进
  improved_code_here

---

## 第四步：解析复杂度深度分析（⚠️ 核心重点）

4.1 语法复杂度量化
计算以下指标：

文法类型判定：

LL(1) / LL(k) / LR(1) / LALR / 无法归类
是否存在左递归、二义性、冲突？
复杂度计数：

终结符数量：___
非终结符数量：___
产生式规则数量：___
最大嵌套深度：___
问题诊断：

 是否存在前瞻冲突（lookahead conflicts）？
 是否需要回溯（backtracking）？
 是否有上下文依赖（context-sensitive）的规则？

4.2 不必要复杂度识别
筛选标准：

语法特性	使用频率	实现成本	复杂度贡献	保留建议
例：可选分号	低	中	增加歧义性	❌ 移除
重点排查：

为了"语法优雅"引入的非必要特性
可通过预处理简化的宏功能
多余的语法糖（syntactic sugar）
过度泛化的元编程特性
4.3 解析器实现成本
评估每个语法特性的实现代价：

### 特性：[特性名称]
- **理论复杂度**：O(?) 时间 / O(?) 空间
- **实现难度**：[简单/中等/困难/极难]
- **维护成本**：[低/中/高]
- **替代方案**：[是否有更简单的实现方式]
